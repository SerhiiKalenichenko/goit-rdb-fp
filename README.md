Проєкт виконано у виробничій логіці:

## Task 1 — Створення схеми та перевірка імпорту

Створено схему `pandemic` та обрано її як активну для подальшої роботи.

Після імпорту даних виконано:
- перегляд перших рядків таблиці `infectious_cases`
- перевірку кількості завантажених записів через `SELECT COUNT(*)`

Це дає змогу:
- переконатися, що імпорт виконано коректно
- зафіксувати базовий обсяг даних для подальшої перевірки ментором

## Task 2 — Нормалізація таблиці infectious_cases

Вихідна таблиця `infectious_cases` містить повторювані атрибути:
- `Entity`
- `Code`

Для усунення дублювання виконано нормалізацію до двох таблиць:

### 1. `entities`
Довідник сутностей, який містить:
- `entity_id` — первинний ключ
- `entity` — назва країни / регіону / сутності
- `code` — код сутності

Реалізовано:
- `PRIMARY KEY`
- `UNIQUE(entity, code)`

### 2. `infectious_cases_normalized`
Нормалізована таблиця фактів, яка:
- зберігає всі показники з вихідної таблиці
- містить зовнішній ключ `entity_id`
- більше не дублює `Entity` та `Code` у кожному рядку

Реалізовано:
- технічний первинний ключ `record_id`
- зовнішній ключ `entity_id`
- видалення дубльованих колонок `Entity`, `Code`
- індекси для `entity_id` і `Year`

### Нормальна форма

Рішення відповідає 3NF:
- 1NF: усі поля атомарні
- 2NF: неключові атрибути залежать від повного ключа
- 3NF: довідникові атрибути `Entity` і `Code` винесено в окрему таблицю, транзитивні залежності усунено

## Task 3 — Аналіз Number_rabies

Для кожної унікальної комбінації `Entity` та `Code` виконано розрахунок:
- середнього значення `AVG`
- мінімального значення `MIN`
- максимального значення `MAX`
- суми `SUM`

Особливості реалізації:
- порожні значення `''` попередньо відфільтровані
- додатково враховано перевірку на числовий формат через `REGEXP`
- значення приведені до числового типу через `CAST`

Результат:
- відсортовано за середнім значенням `avg_number_rabies` у порядку спадання
- обмежено до 10 рядків через `LIMIT 10`

Це відповідає вимогам завдання та унеможливлює викривлення агрегатів через порожні рядки.

---

## Task 4 — Побудова колонки різниці в роках

Для атрибута `Year` побудовано:
- дату першого січня відповідного року через `MAKEDATE`
- поточну дату через `CURDATE()`
- різницю в роках через `TIMESTAMPDIFF`

Особливості реалізації:
- використано `DISTINCT`, щоб не дублювати однакові роки у виводі
- значення року очищено через `TRIM`
- застосовано перевірку формату року через `REGEXP '^[0-9]{4}$'`
- рік приведено до числового типу через `CAST(... AS UNSIGNED)`

Це рішення є коректним і стабільним для роботи з текстово імпортованими CSV-даними.

## Task 5 — Користувацька функція

Створено функцію:

`get_year_difference(input_year INT)`

Функція:
- приймає рік
- будує дату `YYYY-01-01`
- повертає різницю в роках між цією датою та поточною датою

Після створення функцію застосовано до даних таблиці.

Переваги рішення:
- повторне використання логіки
- зменшення дублювання SQL-коду
- чистіший аналітичний запит

## Security Notes

У межах фінального проєкту враховано такі елементи безпеки:

### Контроль доступу
- рекомендовано використовувати окрему роль з мінімально необхідними правами
- для перевірки та аналітики достатньо read-only доступу
- права на `ALTER`, `DROP`, `CREATE FUNCTION` мають бути лише у довіреного користувача

### Захист від SQL injection
- у проєкті відсутній dynamic SQL
- усі запити статичні та контрольовані
- логіка не залежить від небезпечної інтерполяції параметрів

### Цілісність даних
- реалізовано `PRIMARY KEY`
- реалізовано `FOREIGN KEY`
- реалізовано `UNIQUE(entity, code)`

### Аудит
- доцільно логувати зміни схеми, імпорт та DDL-операції
- окремо бажано фіксувати виконання функцій і масових оновлень

### Резервне копіювання
- перед повторною нормалізацією або зміною структури схеми рекомендовано створювати backup
- вихідний CSV-файл потрібно зберігати окремо від робочої БД

### Принципи
- Zero Trust
- Secure by Design
- Least Privilege
